## 关键决策 (产品判断)

1. **Sentence terminator set** — `。`, `？`, `！`, ASCII `?`, `!`. Deliberately
   **excluded** ASCII `.` because Chinese-first scripts routinely contain
   Latin fragments like "v1.2", "www.example.com", "OpenAI's GPT-4.0" where
   the period is not a sentence boundary. If a caller wants English
   sentences to split on periods, they must mark them with full-width `。`.
2. **Consecutive terminators collapse** (`"真的?!"` → one sentence). The
   regex `[^T]*[T]+` eats runs of terminators into the preceding sentence
   rather than emitting empty shards.
3. **Trailing fragment without terminator** is still emitted as its own
   sentence. Otherwise authors would lose content when the final sentence
   forgets a period.
4. **Control markers (`[break]`, `[breath]`, `[long break]`, phonemes) are
   *not* split points and are preserved verbatim in `chunk.text`**. These
   are Fish S2-Pro engine directives, not sentence boundaries. P5 is
   responsible for stripping them before emitting subtitles (per the
   project CLAUDE.md).
5. **`text_normalized = text.strip()`** — P1 never rewrites content. Strip
   only removes leading/trailing whitespace; inner whitespace, punctuation,
   and control markers are preserved.
6. **`char_count = len(text_normalized)`** — Python `len` counts unicode
   code points, which is the right semantics for a Chinese-first product
   (one 汉字 = 1, not 3). Not bytes.
7. **`boundary_hash = sha256(f"{shot_id}|{idx}|{text}")[:16]`** — text is
   the **raw** sentence, not `text_normalized`. Rationale: trimming is
   idempotent, so using raw text gives slightly higher change sensitivity
   without false positives. Truncated to 16 hex chars (64 bits) — plenty
   of collision margin for per-episode uniqueness, and shorter than 64
   chars for readability in logs / the UI.
8. **`shot_id` normalisation** — `int` → `"shot{NN}"` zero-padded to 2.
   Bonus: stringified digits (`"1"`) get the same treatment, so tools that
   JSON-encode ints as strings don't mysteriously break. `bool` is
   explicitly rejected (it's an int subclass, otherwise `True` would
   silently become `"shot01"`).
9. **Re-run strategy: DELETE + bulk_insert** inside a single transaction.
   Chunks are deterministic for a given script, so replaying P1 after an
   edit (or during a retry) produces byte-identical rows and does not
   break referential integrity — takes/stage_runs are cascade-deleted via
   `ON DELETE CASCADE` at the DDL level. This keeps the task trivially
   idempotent and avoids complicated "diff and upsert" logic.
10. **Empty-segments episodes transition to `ready`, not `invalid_state`**.
    Authoring tools save empty drafts; forcing users to re-run after
    adding content is a worse UX than writing 0 chunks. The tradeoff: an
    empty episode in `ready` state cannot actually run P2, but that's the
    downstream stage's validation problem, not P1's.
11. **`P1Context` dataclass for DI** — the task takes `ctx: P1Context`
    carrying a `session_maker` and `storage`. Rationale: Prefect tasks
    should not pull global state; passing the context explicitly keeps
    unit tests honest and lets W3's flow entry point wire in real vs mock
    adapters without monkey patching. `ctx` is not serialised by Prefect
    — it's just a function argument consumed inside the task body.
12. **`DomainError` taxonomy** — `not_found` / `invalid_input` /
    `invalid_state`. Mirrors HTTP semantics so A9-API can map 1:1 when
    surfacing failures. The class is exported from `p1_chunk.py` so the
    flow layer and API layer can `except DomainError` uniformly.
13. **Prefect logger fallback** — the task tries `get_run_logger()` and
    falls back to stdlib `logging` when `MissingContextError` is raised.
    This lets `p1_chunk.fn(...)` be called directly from unit tests
    without a Prefect runtime.

## 放弃的方案

* **Regex using negative lookbehind for "sentence-ending period"** —
  initially tried to include ASCII `.` as a terminator but exclude
  `v1.2` / `www.` / `a.m.` with lookbehinds. Got complicated fast and
  still had false positives on abbreviations. Punted: callers who want
  ASCII-period splits can preprocess their script, or we add a config
  flag in a later wave.
* **Keeping old rows on re-run and computing a diff** — considered so we
  could preserve `takes` across minor edits. Rejected because chunk ids
  are stable for unchanged sentences anyway; and if the user edits a
  sentence, `boundary_hash` changes and downstream *should* invalidate.
  Let cascade delete do its job.
* **Emitting `stage_failed` events inside the task on `DomainError`** —
  rejected because the task's transaction rolls back on exception and the
  event would disappear with it. Better to let the Prefect task state
  machine handle failure visibility (and let A8 add a wrapper flow that
  writes the failure event in a fresh session if needed).
* **Adding a `chunk_count: int` field to `P1Result`** — the existing
  shape `P1Result(episode_id, chunks: list[ChunkInput])` already makes
  this trivially available via `len(...)`, and the spawn prompt
  explicitly said "**不可以**: 改任何已存在的类". Respected.

## 给下游 (W3 集成) 的提示

1. **`P1Context` wiring** — the flow entry point (`run_episode.py` owned
   by A8) needs to construct a `P1Context(session_maker, storage)` and
   pass it as a kwarg: `await p1_chunk(ep_id, ctx=ctx)`. The `session_maker`
   should be an async-sessionmaker bound to the **business** Postgres
   engine, not Prefect's metadata DB.
2. **`DomainError` at the flow level** — catch it and re-raise as a
   Prefect `Failed` state with the `.code` surfaced. A9-API can map
   `not_found` → HTTP 404, `invalid_input` → 422.
3. **Re-run semantics** — P1 always wipes existing chunks for the
   episode before inserting. This means any ongoing P2/P3/P5/P6 work on
   those chunks will be cascade-deleted through `takes` and `stage_runs`.
   The flow orchestrator must ensure no downstream task is in-flight
   when P1 re-runs. Simplest policy: disallow P1 re-run while
   `episode.status in ("running",)`.
4. **`stage_runs` table** — P1 does **not** write a `stage_runs` row.
   `stage_runs` is per-chunk per-stage; P1 is episode-scoped and produces
   chunks rather than consuming them. If the UI wants a "P1 done" signal,
   it should listen for the `stage_finished` event with
   `payload.stage == "p1"`.
5. **Determinism** — `boundary_hash` only depends on `(shot_id, idx, raw
   sentence text)`. If a caller later wants to version-gate on script
   metadata (title / episode config) they should compute their own
   per-episode digest; do not extend `boundary_hash`.
6. **MinIO path constant** — `core.storage.episode_script_key(ep_id)` is
   the only supported way to address the script object. The task uses
   it; callers constructing test fixtures should too.
7. **`segment_type` in metadata** — the task stores `{"segment_type":
   "hook" | "content" | "cta" | ...}` in `chunk.metadata` when the
   script provides it. Downstream consumers can use this to style
   subtitles or pick different voices per segment type, but it is
   optional.
